# Skill: /design

Produce the **detailed design** for a component or feature: the contracts it exposes and consumes, the patterns it uses (named and justified), the data shapes, the error and concurrency model, the failure modes and telemetry, and the test plan. This is the technical blueprint (BoK §IV.1 Plan; Rules of the Road Phase 6) that turns architecture into something implementable one verifiable increment at a time.

**Spine:** Rigor Protocol with **contract due diligence + spikes** at Stage 3. **Authority:** the **LOA pattern catalog** (Part IV) and conformance criteria for AI-integrated components; the **C# Style Guide** for C# shape; the **Testing Strategy** trigger table for the test plan (a *forcing function* — see Stage 3); the **Observability & Instrumentation Standard** (`observability-and-instrumentation.md`) for the telemetry design; and — whenever the component has a user-facing interface — the **UI Archetype Grammar** (`ui-archetype-grammar.md` + `ui-archetype-catalog.md`) to fix the UI archetype (the determinism selector) and the **UI & Interaction Design Standard** (`ui-interaction-design.md`, U1) to make every facet concrete (tokens, states, patterns). **Mode:** Peer Mode for authoring, Adversary Mode at the gate.

## Grounding (first action)
Load the artifacts this design must conform to and treat them as the **authoritative source of truth** (Rigor Protocol Stage 0; BoK §III.1): the spec (`docs/specs/<feature>.md`), the architecture (`docs/architecture.md`), and any existing design in `docs/design/`. Read them before proposing anything, quote the specific spec statements and architectural decisions the design must satisfy, and if the request — or a later artifact — conflicts with them, **stop and surface the drift** (re-spec, or record a deviation) instead of silently diverging (anti-pattern: Specification Drift). Prefer **graph traversal** for this grounding (`knowledge-visualization.md` V15): start from this task's artifact(s) in the knowledge graph and follow the typed edges 1–2 hops (upstream `implements`/`refines`/`depends-on`, downstream `tested-by`/`documents`, and `uses-term` into the glossary), citing the traversal path; a missing edge, stale node, or orphan found here is a finding to surface. Skip this only if the user explicitly tells you not to consult the prior artifacts.

## Input
A spec and/or `docs/architecture.md`, or a prompt for a feature within an existing architecture. If the design would introduce a load-bearing architectural decision, stop and run `/define-architecture` (re-spec, don't quietly expand — anti-pattern: Scope Drift).

## Cast
- **Peers (author together):** Orchestrator, **Patterns Expert** (the established idiom for the problem), **The Simplifier** (the smallest correct version), the relevant **language Developer** (idiomatic shape), **Domain Researcher** (spikes unfamiliar contracts), and the **Distributed Systems** / **Security** architects when the component touches async/messaging or a trust boundary.
- **Adversaries (attack the design at the gate):** **Patterns Expert ⇄ Simplifier** (mutual check — a pattern survives only if it passes both), **Test Architect** (is every behavior verifiable, and is **every triggered Testing Strategy directive** in the plan — hard veto), **Security** (hard veto if identity/secrets/PII), **Distributed Systems** (hard veto if unsafe async/messaging), **SRE** (observability — enforces the Instrumentation Standard: spans, trace-correlated structured logs, stable error codes, RFC 9457 responses).

## Flow (Rigor Protocol, specialized to component shape)

**Stage 0 — Interdict the rush.** Do not write implementation code. Establish the design and its contracts first.

**Stage 1 — OPEN.** Frame the component: its responsibility (one, ideally), its boundaries, what crosses them, the data it owns vs. borrows, the scenarios and the boundary set (empty/null/max/concurrent/malformed/hostile). **Locate it in the delivery phasing**: which vertical slice of the architecture's phasing plan does this component serve, what is real vs. mocked around it at that phase, and which of its seams must be designed as **mock-substitutable contracts** so the slice deploys end-to-end (test- and human-validatable) before its neighbors are built. In an existing codebase, first establish the **internal contracts** (BoK Part V.1): the local conventions, the existing abstractions, the error style, the test patterns — *find the established pattern and conform to it; a competing one is a recorded deviation.*

**Stage 2 — INTERROGATE.** Drill with precise questions: *Clarification* — what exactly is this component responsible for; *Assumption* — what does it assume about its callers, its data, its concurrency; *Cause/Effect* — what pattern fits this problem and what does choosing it imply; *Evidence* — is the bespoke approach reinventing a standard idiom (Patterns Expert) and does the problem actually need the pattern (Simplifier). Name the candidate patterns (GoF, integration/enterprise, LOA Part IV, language idioms) and the rejected ones.

**Stage 3 — EVIDENCE (spikes for the unfamiliar).** For every contract the component consumes — and especially anything unfamiliar, preview, or version-sensitive — run the **Spike Protocol** (read + run) and establish it across the BoK Part III dimensions with cited sources. Confirm the error model, nullability, concurrency, idempotency, and resource lifecycle empirically where cheap. For AI-integrated components, name the LOA patterns in play and check the relevant conformance criteria. Map the component's code shape to the **Testing Strategy trigger table** and **enumerate every triggered directive** — D0 hygiene always, then the *union* of every matching D1–D7 / A1–A6. This is a **forcing function, not a mapping exercise**: a triggered directive that the design does not account for is a gate failure (the Test Architect's hard veto) unless a deviation is recorded with rationale. Then design the component's **telemetry to the Observability & Instrumentation Standard** (`observability-and-instrumentation.md`): the spans it opens, the structured log events and the trace context they carry, the stable **error codes** it can emit, the **RFC 9457** problem responses for any HTTP surface, and the metrics (O1–O13). Telemetry is part of the contract — name it now, do not leave it to implementation. **Failure-mode analysis (mandatory).** Derive the failure modes *from the design choices themselves* — every pattern, boundary, dependency, and concurrency decision you just made introduces specific ways to fail. Walk the categories so the common ones are not missed: inputs (null / empty / malformed / boundary / oversized / duplicate / out-of-order); each external dependency (timeout / unavailable / slow / partial or error response / rate-limited / contract drift); concurrency (race / lost update / ordering / partial failure mid-operation); state and data (staleness / inconsistency across stores / partial write / retry idempotency); resources (exhaustion / leak / backpressure); and time (timeout / expiry / clock skew / retry storm). For **each** failure mode, commit to an explicit, intentional disposition — **prevent** (make it impossible by design), **detect** (name the telemetry that surfaces it — an error code, a log, an alert; ties to the Observability Standard), **mitigate / degrade** (fallback, partial service), **recover** (retry, compensate, roll back), or **consciously accept** (with the rationale and the residual risk written down). No failure mode is left implicit: the design states, for each, *how it is addressed or why it is accepted*. Every mode dispositioned as handled and load-bearing becomes a negative/error-path entry in the test plan — these are exactly the cases the urge to complete skips. **Adversarial analysis — STRIDE-lite (mandatory).** Failure-mode analysis covers what goes wrong *by accident*; this covers what goes wrong *on purpose*. Identify the component's **trust boundaries** (every place data or control crosses from a less-trusted to a more-trusted context: public endpoints, queue consumers, file/config ingestion, inter-service calls, admin surfaces), then walk **STRIDE** per boundary — **S**poofing (who could claim a false identity here?), **T**ampering (what data in transit/at rest could be altered?), **R**epudiation (what actions lack attributable evidence? ties to the Observability Standard's audit events), **I**nformation disclosure (what could leak — including through error messages and logs, O-series PII rules), **D**enial of service (what can be exhausted or flooded?), **E**levation of privilege (how could lesser rights become greater?). For **each** identified threat, commit to a disposition exactly as for failure modes — **mitigate** (named control: authn/authz check, validation, signing, rate limit, least privilege), **transfer** (an upstream gateway/platform control, *named*, not assumed), or **consciously accept** (rationale + residual risk written down). Misuse cases for mitigated threats become **negative security tests** in the test plan (the forged token, the tampered payload, the over-privileged caller, the flood). The **Security & Identity Architect holds the hard veto** here, exactly as the Test Architect does for test directives. **Privacy analysis — LINDDUN-lite (mandatory when the component touches personal data).** If any data flow involves a person (directly or by inference), walk **LINDDUN** over it — **L**inkability, **I**dentifiability, **N**on-repudiation, **D**etectability, **D**isclosure (including *through logs and telemetry* — the O-series no-PII rules are a privacy control, verify them here), **U**nawareness, **N**on-compliance (retention, lawful basis, rights paths) — and disposition each finding (mitigate with a named control / accept with rationale + residual risk), recording the data categories touched, retention, and the rights-path mechanism. Findings go in the design's **Privacy analysis (LINDDUN-lite)** section (same table discipline as STRIDE — the repo-level Privacy Review rolls these up mechanically). The **Privacy & Data Governance lens holds the veto** on unanalyzed personal-data flows. If the component touches no personal data, state that in one line — an explicit negative, not an omission. **UI design — interface excellence (mandatory when the component has a user-facing interface, UI Standard U19).** If the component renders a screen, component, flow, page, CLI output, or voice turn, specify the interface, not just the system beneath it: the **medium(s)** and authoritative platform guidelines (U2); the **token references** the UI uses, or the new primitive/semantic/component tokens it defines — no arbitrary values (U3–U5); the **key screens/flows** with their visual hierarchy and single focal point (U6); the **complete state set per component** — default / hover / focus / active / disabled / **loading** / **empty** / **error** / **success** plus first-run and overflow (U9, the most common place polish is missing); the **motion** intent with reduced-motion behavior (U10); the **real UI copy** for the load-bearing strings, in the product's voice — no lorem (U11); the **medium adaptation** (how it changes across the declared media); and the **accessibility (WCAG 2.2 AA) and performance-budget** approach (U16–U17). For **AI-facing** UIs, additionally name the applicable **HAX guidelines** (G1–G18 across initially/during/when-wrong/over-time) and the **Shape-of-AI patterns** in play — Wayfinders to defeat the blank canvas, Tuners to shape requests, **Governors** for oversight on consequential actions (action plan before execution, stream-of-thought, controls to steer, verification before irreversible steps), Trust builders (disclosure, caveats, citations, provenance), Identifiers — and design the **wrong answer and uncertainty as first-class states** (effortless correction/regeneration, honest confidence, graceful degradation) (U13–U15). Honor the established pattern for each job unless a deviation is recorded with rationale (U12). The **UX & Accessibility lens co-owns the UI design and holds the accessibility hard veto** (U16). Use the `frontend-design` skill as the hands-on craft companion for web/component media. Maintain the confidence ledger.

**Stage 4 — DISCONFIRM (the gate).** Switch to Adversary Mode. The Patterns Expert and Simplifier check each other (no cargo-cult pattern, no needless abstraction); the Test Architect demands a verification path for every behavior the design promises and attacks coverage gaps with falsifying inputs; Security and Distributed Systems apply their hard vetoes via disconfirming claims; the SRE attacks the **failure-mode analysis** — is every mode dispositioned, is each "accept" actually justified, is each handled mode detectable from telemetry alone and backed by a test? Probe for the *missing* failure mode the analysis didn't list. The **Security & Identity Architect attacks the adversarial analysis** the same way — is every trust boundary identified, is every STRIDE threat dispositioned, is each "transfer" a *named* control rather than an assumption, does every mitigation have its negative security test? An unexamined boundary is a veto. Resolve or record every veto; authors do not self-clear.

**Stage 5 — CONVERGE.** Write the design to `templates/design.template.md`: the contracts (with source citations at the boundaries), the named-and-justified patterns, the data shapes, the error/concurrency model, the **failure-mode analysis** (each mode → its disposition: prevent / detect / mitigate / recover / accept, with accepted ones carrying rationale + residual risk), the **adversarial analysis** (trust boundaries; each STRIDE threat → mitigate / transfer / accept, with misuse-case tests for mitigations), the **privacy analysis** (LINDDUN findings → dispositions, or the explicit no-personal-data line) and the **telemetry design** (spans, structured-log events with trace context, stable error codes, RFC 9457 responses for HTTP, and metrics — per the Observability Standard, O1–O13), and the **test plan** (every Testing Strategy directive the triggers select, D0 included — not a sample; every handled failure mode appears as a negative/error-path test). For UI components, also write the **UI design**: token references (or new tokens), the key screens/flows with hierarchy, the **complete per-component state set** (incl. empty/loading/error), motion intent, the real UI copy, medium adaptation, the accessibility + performance approach, and — for AI UIs — the named HAX guidelines and Shape-of-AI patterns (U13–U20). State the residual risk and the flagged unknowns. Name patterns so they will appear in code comments (`// Pattern: … (LOA 3.2)`).

**Close with the status table (mandatory).** End the response with:

| | |
|---|---|
| **Completed** | the component(s) designed in this run |
| **Remaining** | the components/phases still undesigned (per the architecture's phasing plan) |
| **Best next action** | the single concrete next step (typically `/implement` of this design, or the next slice's `/design`) |

## Output artifact
`docs/design/<component>.md` (from `templates/design.template.md`): responsibility, contracts (exposed + consumed, sourced), patterns (named + justified), data shapes, error/concurrency model, **failure-mode analysis (mode → disposition)** + **adversarial analysis (boundary → STRIDE threat → disposition)** + telemetry, test plan (Testing Strategy directives), conformance notes, flagged risks, gate record.

## Definition of done (exit gate)
- [ ] Single clear responsibility; boundaries and contracts named.
- [ ] **Placed in the delivery phasing**: the vertical slice this component serves named; mock-substitutable seams identified so the slice deploys end-to-end at its phase.
- [ ] Local conventions established and conformed to (or deviation recorded).
- [ ] Every consumed contract established; **unfamiliar ones spiked**; sources cited at the boundary.
- [ ] Every pattern **named and justified**; survives both Patterns Expert and Simplifier.
- [ ] **Failure-mode analysis complete**: modes derived from the design choices across the input/dependency/concurrency/state/resource/time categories, each with an explicit disposition (prevent / detect / mitigate / recover / accept); accepted modes carry rationale + residual risk; handled modes appear in the test plan (SRE + Test Architect).
- [ ] **Adversarial analysis complete**: trust boundaries identified; STRIDE walked per boundary; every threat dispositioned (mitigate / transfer-named / accept-with-rationale); mitigations carry negative security tests in the test plan (Security & Identity Architect's hard veto).
- [ ] **Privacy analysis complete**: LINDDUN walked over every personal-data flow with dispositions, data categories, retention, and rights paths — or the explicit no-personal-data line (Privacy & Data Governance veto).
- [ ] **UI design complete (if the component has an interface)**: medium + platform guidelines named; token references / new tokens (no arbitrary values); key screens with hierarchy; **complete per-component state set incl. empty/loading/error**; motion + reduced-motion; real in-voice copy; medium adaptation; WCAG 2.2 AA + performance budget; and for AI UIs the named HAX guidelines and Shape-of-AI patterns with wrong-answer/uncertainty designed (UI Standard U13–U20; UX & Accessibility lens's accessibility veto).
- [ ] **Rollups current**: `docs/security/threat-model.md` and `docs/security/privacy-review.md` exist, link `documents` to this design, and their generated registers were refreshed via `docs-graph.py rollup`.
- [ ] **Every** triggered Testing Strategy directive enumerated in the test plan (D0 + the union); none skipped without a recorded deviation; every promised behavior has a verification path (Test Architect — forcing function, hard veto).
- [ ] **Telemetry designed to the Observability Standard**: spans, trace-correlated structured-log events, stable error codes, RFC 9457 responses (HTTP), and metrics named (O1–O13); load-bearing telemetry has a planned test (SRE).
- [ ] Hard vetoes (Security, Distributed Systems, Test Architect) resolved or deviated; authors did not self-clear.
- [ ] Confidence ledger updated; residual risk stated.
- [ ] **Status table emitted**: completed / remaining / best next action.

## Documentation & discoverability (last action)
Per the **Knowledge Visualization & Docs Explorer Standard** (`knowledge/knowledge-visualization.md`, the Discoverability Mandate V10): after producing the design (`docs/design/<component>.md`), **write the artifact's frontmatter** (the record, V2: id, title, type, status, **owner**, phase, tags, **typed links** per the relation registry, **review-by** per the type's SLA, and a real 1–3-sentence summary) and **sync the derived `docs/docs-index.js`** by running the script bundle — `python3 docs/ai-forward-pack/scripts/docs-graph.py derive` (and `flag --changed <id> --reason …` for V16 propagation) — never by generating ad-hoc scripts (V18); frontmatter wins wherever the two disagree. Ensure `docs/index.html` (the Docs Explorer) exists — instantiate it from `templates/docs-explorer.template.html` if missing — and verify the new entry has at least one typed link into the graph (an orphan is a finding). Index and diagrams land **in the same change** as the content (V11). **Update the repo security/privacy rollups:** ensure `docs/security/threat-model.md` and `docs/security/privacy-review.md` exist (stub from `templates/threat-model.template.md` / `templates/privacy-review.template.md` if missing) and each carries a `documents` link to this design (that link is what makes V16 flag them automatically on future changes); refresh their generated registers with the script bundle — `python3 docs/ai-forward-pack/scripts/docs-graph.py rollup --heading "Adversarial analysis (STRIDE-lite)" --type design` and `… rollup --heading "Privacy analysis (LINDDUN-lite)" --type design` — and update the accepted-risk registers by hand for any new `accept`. **Propagate impact (V16):** if this change is material (a contract, requirement, decision, public shape, or proven claim changed), traverse the **inbound** edges and push `review-suggested: { by, on, reason }` into each inbound neighbor's frontmatter (and its index entry) in this same change. **Capture session exhaust (V17):** any mid-session decision, discovered assumption, or resolved question below ADR weight becomes a linked **decision note** (`docs/notes/`, from `templates/decision-note.template.md`) before close — promote to an ADR if it later bears load. Work that is not discoverable in the Explorer is not done.
