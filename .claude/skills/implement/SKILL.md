# Skill: /implement

Build the feature/component as a sequence of **small, independently verifiable increments**, each proven correct against the spec's boundary set — not against a green checkmark (BoK D1). Implementation is the micro-loop: **red → green → refactor** (BoK §IV.2), worked as a *pair* between the language Developer and the Test Architect, then handed to an adversarial pre-merge review that the author cannot self-approve (BoK D3).

**Spine:** Rigor Protocol, with Stage 3 = execution (the tests *are* the evidence) and Stage 4 = the Test Architect's hard veto. **Authority:** the **Testing Strategy** (`testing-strategy.md`) is the normative, **mandatory** decision procedure for what to test — triggered directives are *applied*, not chosen (a forcing function); the **Observability & Instrumentation Standard** (`observability-and-instrumentation.md`) governs logging, tracing, error codes, and error responses; the **C# Style Guide** / language craft (BoK Part VII) for how code is written; the **LOA** idiom map for AI-integrated code. **Mode:** Peer Mode (pairing) for building, Adversary Mode for review.

## Grounding (first action — before any code)
Load the artifacts this implementation must conform to and treat them as the **authoritative source of truth** (Rigor Protocol Stage 0; BoK §III.1): the design (`docs/design/<component>.md`) **and** its spec (`docs/specs/<feature>.md`), the architecture (`docs/architecture.md`), and the existing code and tests in the area you are touching. **Re-read them now — do not work from memory of them.** Quote the specific spec statements and design contracts each increment must satisfy, and trace every task back to one. If the request, the existing code, or your emerging implementation conflicts with the spec or design, **stop and surface the drift** (reconcile against the spec, or record a deviation per Rules of the Road §4) rather than quietly building something else — this is the failure mode that produces spec-to-implementation drift. Prefer **graph traversal** for this grounding (`knowledge-visualization.md` V15): start from this task's artifact(s) in the knowledge graph and follow the typed edges 1–2 hops (upstream `implements`/`refines`/`depends-on`, downstream `tested-by`/`documents`, and `uses-term` into the glossary), citing the traversal path; a missing edge, stale node, or orphan found here is a finding to surface. Skip this grounding only if the user explicitly tells you not to consult the prior artifacts.

## Input
A design (`docs/design/<component>.md`) or — for small, low-risk work — a prompt. If implementing from a prompt would touch security/identity/data/contracts/money/concurrency, raise the tier and run `/design` first (Rules of the Road §0.2; the higher tier wins when unsure).

## Cast
- **Peers (pair to build):** the relevant **language Developer** (C#/Rust/Python — idiomatic shape) ⇄ **Test Architect** (testable slices). **Domain Researcher** for any just-in-time spike a contract turns out to need. **Orchestrator** keeps the thread and the Proof Pack.
- **Adversaries (attack pre-merge):** **Test Architect** (hard veto — no unverifiable claim, and no triggered Testing Strategy directive left unapplied), the **language Developer** (idiom/style/async correctness), **SRE** (enforces the Instrumentation Standard — structured logs in the OTel model with trace context, stable error codes, RFC 9457 responses — plus resource bounds), and the relevant architects if the increment touches their lens. *Generated code is a proposal until reviewed; the producer does not certify the producer* (BoK §VII.7).

## Flow (Rigor Protocol, specialized to building)

**Stage 0 — Interdict the rush.** Do not emit a wall of code. Decompose first; build one verifiable slice at a time.

**Stage 1 — OPEN (plan tasks).** Decompose the design into small tasks, each independently verifiable, each tracing to a spec statement (Engineering Governance §1). Re-establish the boundary set per task (empty/null/max/concurrent/malformed/hostile). In an existing codebase, confirm local conventions before writing (BoK Part V.1). For changing *existing* untested behavior, **characterize before you change** (BoK §V.2): pin current behavior with characterization tests first.

**Stage 2 — INTERROGATE (shape the tests).** For each task, map its code shape to the **Testing Strategy trigger table** (T1–T14) and take the **union** of every matching directive — this is mandatory: a triggered directive is *applied* or a deviation is recorded with rationale, never silently dropped. Apply **D0 hygiene** to every test unconditionally. Ask the Test Architect's questions up front: for each behavior the design promises, what test proves it, and what input makes that test *fail*? A test with no failing input asserts nothing. **Failure-mode analysis (mandatory).** Carry the design's failure-mode analysis forward and extend it with the failure modes your *implementation choices* introduce — the specific libraries, the actual error handling, the concurrency primitives, the parsing/serialization. Walk the same categories (inputs / each dependency / concurrency / state & data / resources / time) so the common error cases are not skipped. For **each** failure mode, the design's disposition must become real in the code — a guard, a typed exception with a stable error code, a fallback, a retry or compensation, a validated boundary — or be consciously accepted with a recorded rationale. Turn every handled mode into a **failing test first** (the null, the timeout, the malformed payload, the concurrent writer, the dependency that errors): these negative and error-path tests are precisely the ones the urge to complete skips, and here they are not optional. **Carry the adversarial analysis forward the same way (STRIDE-lite).** The design's threat dispositions must become real in the code — the authn/authz check, the input validation at the trust boundary, the signed or constant-time comparison, the rate limit, the least-privilege scope — and implementation choices that *create new boundaries or capabilities* (a new endpoint, a deserializer, a file path from input, a subprocess, dynamic SQL) get the STRIDE walk too. Every mitigation gets its **failing negative security test first** (the forged/expired token, the tampered payload, the path-traversal name, the injection string, the over-quota flood). A design-mitigated threat with no enforcing code or no negative test fails the **Security & Identity Architect's hard veto**, exactly as a triggered test directive fails the Test Architect's. **Carry the privacy analysis forward too (LINDDUN-lite):** the design's privacy dispositions become real — the redaction in the structured-log enricher, the retention job, the erasure path, the consent/purpose check — and every mitigation gets its **failing privacy test first** (the no-PII-in-telemetry test with a sample value, the erasure that actually deletes, the export that contains exactly the inventory's categories). Implementation choices that *create new personal-data handling* (a new log field, a cache, an analytics call) get the LINDDUN walk. Unenforced privacy dispositions fail the **Privacy & Data Governance veto**. **Build the UI to the system and prove its states (UI Standard U20, when the component has an interface).** **Honor the spec's UI Archetype Signature** (`ui-archetype-grammar.md` G11): resolve the archetype's codegen descriptor (`ui-archetype-catalog.md`) and apply every facet's rule — the routing/temporal model (`Arch`/`Pacing`: guarded steps, gated progression, no deep-linking past an incomplete gate), the data posture (`Sync`: optimistic-with-rollback / local-first / async-pipeline-with-narrated-wait / server-strict), and the feel (`Feedback`/`Motion`/`Transition`) — so the build is the *same archetype* the spec selected, not a generic regression to the mean. Then build against the **design tokens** — no raw hex, magic numbers, or one-off durations in component code (U3); implement **every specified component state** including the ones the urge to complete skips — **empty, loading, error**, plus disabled/focus/first-run/overflow (U9); honor the platform HIG and the **responsive/adaptive** behavior per medium (U2); wire **motion** with `prefers-reduced-motion` support (U10); use the **real, in-voice UI copy** from the design, not lorem (U11); and meet the **accessibility (WCAG 2.2 AA)** and **performance-budget** floors (U16–U17). Prove these as tests written **red-first**, exactly like security tests: the keyboard-only path, a contrast/labeling check, the empty/error-state render, and a performance-budget threshold assertion (perceived load / interaction latency / frame-rate / payload). For **AI-facing** UIs, make the named HAX guidelines and Shape-of-AI patterns real — the disclosure that content is AI, the correction/regeneration affordance, the action-plan-before-irreversible-action, the citations/provenance — each with its test. A triggered-but-unmet UI directive fails the gate: the **UX & Accessibility lens's veto** for inclusion and state completeness, the Test Architect's for coverage.

**Stage 3 — EVIDENCE (TDD: red → green → refactor).** Work the pair loop, one slice at a time:
- **Red.** Write a failing test that encodes one slice of spec behavior (not the implementation's incidental shape). **Observe it fail** — a test never seen red may be a tautology.
- **Green.** Write the minimum code to pass it. Conform to the Style Guide and the LOA idiom map as you write (`TimeProvider`, `IHttpClientFactory`, Polly v8 `ResiliencePipeline`, `CancellationToken` threaded, no `async void`, no `.Result`; nullable enabled; typed/narrow exceptions). As you write, **instrument to the Observability Standard** (`observability-and-instrumentation.md`): open a span per unit of work; emit **structured** logs (message template + named fields, never interpolated strings) that carry **trace context** (`trace_id`/`span_id` from the active `Activity`); raise every failure with a **stable error code** and, on any HTTP surface, an **RFC 9457** problem response; keep secrets and PII out of all telemetry (O1–O13). Name patterns in comments (`// Pattern: … (LOA 3.2)`).
- **Refactor.** Improve structure with the tests as a safety net; keep the suite green in small steps; never mix a refactor and a behavior change in one silent commit (BoK §V.2).
- If a contract turns out to be unfamiliar, **spike it** (`spike-protocol.md`) before coding against it, then make the spiked behavior a real test.

Reach for the wider toolkit when cost-of-error warrants (Testing Strategy): property-based tests for wide input domains, contract/schema tests at boundaries (D5/D6), characterization tests for legacy, mutation testing to answer *"does the suite fail when the code is wrong?"*. For AI-integrated code: split deterministic structure checks from probabilistic content evals (A1–A6); treat prompt/schema/tool-description changes as contract changes with a regression gate (A6).

**Stage 4 — DISCONFIRM (pre-merge review).** Switch to Adversary Mode. The Test Architect attacks coverage: mutation sense (would a flipped `+`/deleted guard/`&&`→`||` be caught?), boundary coverage, tautologies, mock fiction, flaky/non-deterministic tests, probabilistic exact-match. **This is a hard gate: any triggered Testing Strategy directive that was not applied — and not deviation-recorded — fails the review.** The Test Architect and SRE also confirm that **every failure mode from the analysis is handled in code and proven by a negative/error-path test** (or its acceptance is recorded with rationale), and probe for the failure mode that was missed. Run mutation tooling where configured and address survivors. The language Developer attacks idiom and async correctness. The SRE attacks the instrumentation against the Observability Standard: are logs structured and trace-correlated, does every failure carry a stable error code, do HTTP errors return RFC 9457, is any secret or PII leaking into telemetry (O1–O13)? Resolve the Test Architect's **hard veto**; the author does not clear it.

**Stage 5 — CONVERGE (verify & report).** Demonstrate correctness against the spec's success criteria and the boundary set — not merely a green suite. Produce the **Proof Pack** (Rules of the Road §3.1): one row per correctness claim — claim, evidence (test name/assertion/run output), source/`file:line`, oracle (why it can fail), red-observed (seen to fail before the fix), confidence, residual risk. Re-ground against the spec (BoK §VI.2): confirm no drift, no contradicted earlier decision. Record any deviations (Rules of the Road §4).

**Close with the status table (mandatory).** End the response with:

| | |
|---|---|
| **Completed** | the tasks/slices implemented and proven in this run |
| **Remaining** | the tasks or phases still unimplemented (per the design and the architecture's phasing plan) |
| **Best next action** | the single concrete next step (the next task, the next slice's `/design`, or ship) |

## Output artifacts
- The code and its tests (tests ship with code, traceable to spec statements, covering the boundary set).
- The **Proof Pack** (`templates/proof-pack.template.md`) and gate records, committed with the change for T1/T2.
- Updated confidence ledger; any ADR/deviation notes.

## Definition of done (exit gate)
- [ ] Every task traces to a spec statement; boundary set covered, not just the happy path.
- [ ] **Failure-mode analysis carried through**: every failure mode from the design (plus those the implementation introduced) is handled in code and proven by a negative/error-path test, or its acceptance is recorded with rationale + residual risk.
- [ ] Every test invokes the focal behavior, has a meaningful assertion, is deterministic, and was **seen red before green** (or justified via mutation/fault injection).
- [ ] **All** triggered Testing Strategy directives applied as a union (mandatory — none dropped without a recorded deviation); D0 hygiene on every test; mutation/eval thresholds wired or recorded as residual risk (Test Architect hard veto).
- [ ] **Instrumented to the Observability Standard**: a span per unit of work, structured logs carrying trace context, stable error codes on failures, RFC 9457 problem responses on HTTP errors, no secrets/PII in telemetry (O1–O13); load-bearing telemetry covered by a test (SRE).
- [ ] Style Guide / LOA idiom conformance; patterns named in comments.
- [ ] Test Architect's hard veto cleared; pre-merge review passed; author did not self-certify.
- [ ] **Proof Pack** complete (Oracle, Red-observed, Confidence filled); residual risk stated; deviations recorded.
- [ ] **Status table emitted**: completed / remaining / best next action.
- [ ] **UI built to the system (if the component has an interface)**: tokens only (no arbitrary values); **every specified state implemented incl. empty/loading/error**; platform HIG + responsive/adaptive behavior; motion with reduced-motion; real in-voice copy; **WCAG 2.2 AA and the performance budget met and proven red-first** (keyboard path, contrast/labeling, empty/error render, perf threshold); AI-UI HAX/Shape-of-AI patterns made real with tests (UI Standard U20; UX & Accessibility lens's veto).

## Documentation & discoverability (last action)
Per the **Knowledge Visualization & Docs Explorer Standard** (`knowledge/knowledge-visualization.md`, the Discoverability Mandate V10): after producing the Proof Pack and any indexed source modules, **write the artifact's frontmatter** (the record, V2: id, title, type, status, **owner**, phase, tags, **typed links** per the relation registry, **review-by** per the type's SLA, and a real 1–3-sentence summary) and **sync the derived `docs/docs-index.js`** by running the script bundle — `python3 docs/ai-forward-pack/scripts/docs-graph.py derive` (and `flag --changed <id> --reason …` for V16 propagation) — never by generating ad-hoc scripts (V18); frontmatter wins wherever the two disagree. Ensure `docs/index.html` (the Docs Explorer) exists — instantiate it from `templates/docs-explorer.template.html` if missing — and verify the new entry has at least one typed link into the graph (an orphan is a finding). Index and diagrams land **in the same change** as the content (V11). **Update the repo security/privacy rollups:** if this implementation changed or added any STRIDE/LINDDUN finding, refresh the generated registers in `docs/security/threat-model.md` and `docs/security/privacy-review.md` via `docs-graph.py rollup` (both headings) and update the accepted-risk tables. **Propagate impact (V16):** if this change is material (a contract, requirement, decision, public shape, or proven claim changed), traverse the **inbound** edges and push `review-suggested: { by, on, reason }` into each inbound neighbor's frontmatter (and its index entry) in this same change. **Capture session exhaust (V17):** any mid-session decision, discovered assumption, or resolved question below ADR weight becomes a linked **decision note** (`docs/notes/`, from `templates/decision-note.template.md`) before close — promote to an ADR if it later bears load. Work that is not discoverable in the Explorer is not done.

**Audit (last action).** Append an audit-log entry for this run — `python3 docs/ai-forward-pack/scripts/audit-log.py append --shortname "implement-<subject>" --session "<id>" --skill implement --kind skill --prompt "<the prompt, verbatim>" --summary "<what shipped + Proof Pack>" --artifact <changed paths>` — per the Audit Mandate (`knowledge/audit-and-change-log.md`, AL5); if a decision was made mid-build (not merely executed), add a change-log entry too (`audit-log.py change`, CL1). A run that left no trace in `docs/audit/` is, like an un-indexed artifact, not done.

**Handoff:** → ship, or → `/investigate` if a defect surfaces.
