# Skill: /document

Produce a **documentation bundle** that is *true to the code* and *current* — an API reference in JavaDoc style, the four diagram families (sequence, class, layered architecture, component), a prose architecture overview, a close-up HTML view — **and maintain the repo's knowledge graph**: `/document` is the bulk maintainer of the Docs Explorer index (`knowledge/knowledge-visualization.md`), performing the **full-sweep regeneration** of `docs/docs-index.js` across *every* artifact type (specs, architecture, ADRs, designs, investigations, proof packs, knowledge, API docs), where the other skills update only their own entries incrementally. The bundle and the index are maintained artifacts with a freshness contract, not a one-time dump.

**Spine:** runs the Rigor Protocol (`knowledge/rigor-protocol.md`) on the *codebase as the subject of documentation* — the evidence is the source itself. **Authority:** the **Knowledge Visualization & Docs Explorer Standard** (`knowledge/knowledge-visualization.md`, V1–V18) governs the graph model, the index schema, the UML/C4 diagram discipline, and the projections. **Mode:** Peer Mode to author, Adversary Mode to verify the docs against the code (`knowledge/collaborative-personas.md`). **Steward:** the **Documentation Steward** owns this skill, the index, and the after-commit automation.

## Tooling — the docs script bundle (V18)
The mechanical work of this skill runs through **`docs/ai-forward-pack/scripts/docs-graph.py`** (stdlib-only Python; no ad-hoc scripts at prompt time): `inventory` for the Stage-1 graph scan (invalid frontmatter, dangling links, unregistered rels, orphans, stale, flagged, index drift — JSON), `derive` for the Stage-5 full derivation sweep, `freshness --gate warn|fail` for the gate, `flag`/`clear-flag` for V16 propagation, `stub` to scaffold schema-correct artifacts, `rollup --heading … --type design` to regenerate the threat-model / privacy-review registers from the designs' own tables, and `snapshot` to append the graph-health record (the governance trend — falling decision-note volume or rising stale counts is the early signal the discipline is decaying). The skill's judgment work — reading code, writing summaries, extracting typed links, verifying diagrams — stays with the agent; the mechanics stay in the tool. A recurring mechanical need is met by **extending the bundle** (with tests), not by regenerating logic per session.

## Grounding (first action)
Documentation describes **what was actually built**, so the code is the ground truth — but load the spec and design too and treat the set as authoritative (Rigor Protocol Stage 0; BoK §III.1): the spec, the design, and the code being documented. Where the code and the spec/design disagree, the documentation reflects the code **and surfaces the discrepancy** as a finding (stale docs that describe a plan never shipped are worse than no docs). Prefer **graph traversal** for this grounding (`knowledge-visualization.md` V15): start from this task's artifact(s) in the knowledge graph and follow the typed edges 1–2 hops (upstream `implements`/`refines`/`depends-on`, downstream `tested-by`/`documents`, and `uses-term` into the glossary), citing the traversal path; a missing edge, stale node, or orphan found here is a finding to surface. Skip this grounding only if the user explicitly tells you not to consult prior artifacts.

## Input
A repository, a module, or a path (default: the whole repo). Optionally a scope hint (`--changed` to document only what a commit touched). For an existing codebase with no docs, this bootstraps the full bundle; thereafter it updates incrementally.

## Cast
- **Peers (author together):** **Documentation Steward** (lead), the relevant **language Developer** (doc-comment conventions for C#/Rust/Python/mobile/desktop), **Patterns Expert** (name the patterns the diagrams should make legible), **Enterprise Architect** (the layered/component diagrams are the architecture of record).
- **Adversaries (verify at the gate):** **Documentation Steward** in Adversary Mode (truth, coverage, diagram fidelity, freshness), **The Simplifier** (no doc bloat — document the public surface, not trivia — soft veto), **Test Architect** (examples must compile/run as written).

## Flow (Rigor Protocol, specialized to the code)

**Stage 0 — Interdict the rush.** Do not paraphrase what the code *should* do — describe what it *does*. No invented examples, no aspirational diagrams. If a doc comment is wrong, the doc is wrong; flag it rather than copy it.

**Stage 1 — OPEN.** Inventory **two surfaces**. *The code:* enumerate the public/exported types and members; the modules and their dependency edges; the key runtime flows worth a sequence diagram; the architectural layers (map to the LOA tiers if the project uses them). *The knowledge graph:* run `docs-graph.py inventory` and read its JSON — it enumerates **every artifact under `docs/`** — specs, architecture, ADRs, designs, investigations, proof packs, knowledge bases, the glossary, API docs — reading each artifact's **frontmatter** (the record, V2) and the current derived `docs/docs-index.js`; note files with missing or schema-invalid frontmatter, entries whose files are gone, **orphan nodes** (no typed links, V10.4), **stale nodes** (`review-by` past, V13), **flagged nodes** (`review-suggested` pending, V16), decision notes in `docs/notes/` (V17), the **security/privacy rollups** (`docs/security/threat-model.md`, `docs/security/privacy-review.md`) and whether their generated registers lag the designs, and frontmatter↔index disagreements (frontmatter wins). Decide scope (full vs `--changed`). List what must be read to document each.

**Stage 2 — INTERROGATE.** For each artifact, ask the precise question: *for this type* — what is its responsibility, its contract, its failure modes; *for this flow* — what is the real call order and where are the awaits/errors; *for this module graph* — what actually depends on what (read the imports, don't guess); *for the layers* — which tier does each component sit in and is any dependency pointing the wrong way.

**Stage 3 — EVIDENCE (read the source; this is the heart of /document).**
- **API reference (JavaDoc-style), per public type/member:** summary, parameters, returns, throws/errors, examples, remarks, see-also — *extracted from the code's own doc comments* (C# XML-doc, JSDoc/TSDoc, Python docstrings, KDoc, Swift doc comments). Where a public member has no doc comment, record it as a coverage gap (do not fabricate prose). Examples must be real and runnable.
- **Diagrams as Mermaid** (renders in GitHub markdown and in the HTML view):
  - **Sequence** — for each key flow, the real ordered interactions across components (derive from the call graph; mark async/awaits and error paths).
  - **Class** — the real type relationships (inheritance, composition, interface implementation, key fields/methods) for the documented module.
  - **Layered architecture** — the LOA capability tiers (T0–T4) or the project's layers, with components placed and dependency direction shown.
  - **Component** — modules/packages as nodes, real dependency edges, external systems at the boundary.
- **Typed links (the graph's edges, V2):** while reading, extract the traceability relations the index will carry — which design each module `implements`, which spec the design `implements`, which ADR a component `depends-on`, which proof pack `tested-by` covers which design, what `supersedes` what. Links are typed (`rel` mandatory); each is **Verified** against the artifact text, never guessed.
- **Diagram discipline (V8):** the architecture overview follows **C4 zoom** — Context and Container for the system view (Component only where it earns its place; pick the level for the audience); sequence for interactions, flowchart/activity for branching flows, class/object for structure, state for lifecycles. `docs/index.md` is written as a **Map of Content** (V3): a curated link layer over the graph, not a folder mirror.
- Label every diagram and every API claim **Verified** (traced to source / example executed) / **Inferred** / **Flagged** in the bundle's confidence note.

**Stage 4 — DISCONFIRM (the gate).** Switch to Adversary Mode (Documentation Steward): walk each diagram node/edge against the code and each example against a compile/run; the Simplifier strikes anything documenting trivia or repeating the obvious; the Test Architect rejects any example that would not run. **Every public-surface gap and every doc-vs-code contradiction is recorded.** The author does not clear its own findings.

**Stage 5 — CONVERGE — emit three surfaces.**
- **Markdown bundle** under `docs/` (structure per `templates/documentation-bundle.template.md`): `docs/api/*.md`, `docs/diagrams/*.md` (Mermaid), `docs/architecture.md`, `docs/index.md` (the MoC), and a `docs/_meta.json` recording the documented commit + coverage + confidence.
- **The map — full derivation sweep (V2/V10).** Run **`docs-graph.py derive`** — it derives `docs/docs-index.js` from the artifacts' **frontmatter** for the whole repo: one entry per artifact (every type — spec, architecture, adr, design, investigation, proof-pack, knowledge, glossary, api, doc), each entry the artifact's frontmatter (id, title, type, status, owner, phase, tags, typed links, review-by, summary) plus path and Mermaid diagram sources; repair or flag missing/invalid frontmatter at the source (the file) rather than patching only the index; **remove entries for deleted files**; bump `generated`/`generator`. Ensure the **Docs Explorer** at `docs/index.html` exists (instantiate from `templates/docs-explorer.template.html` if missing) and loads the index. **Refresh the security & privacy rollups:** ensure `docs/security/threat-model.md` and `docs/security/privacy-review.md` exist (from their templates), each linking `documents` to the architecture and to every (personal-data-touching, for privacy) design; regenerate their registers via `docs-graph.py rollup` (the STRIDE heading and the LINDDUN heading); verify the trust-boundary map matches the C4 Context view; record designs *missing* their STRIDE/LINDDUN sections as upstream gate failures in each doc's Gaps section; clear any `review-suggested` flags on the rollup docs that this refresh resolves (`docs-graph.py clear-flag`). The **Security & Identity Architect** reviews the threat model; the **Privacy & Data Governance lens** reviews the privacy review. Run `docs-graph.py snapshot` — every /document run appends the health record that makes governance drift measurable over time. Report remaining **orphan**, **stale** (`review-by`-past, V13), and **review-suggested** (V16) nodes as findings — honest projections (V12) over silent omissions; the Explorer's health view shows all three. Long-unresolved `review-suggested` flags are escalations to the flagged artifact's owner, not noise to clear.
- **The close-up — browsable bundle view** at `docs/_site/index.html` from `templates/doc-viewer.template.html`: a self-contained single file that renders the bundle's markdown + Mermaid with navigation, opens via `file://` (content embedded). The Explorer is the *map* of everything; `_site` is the *close-up* of the documentation bundle — the Explorer's `doc` entries link to it.

## After-commit freshness (the Documentation Steward's automation)
The bundle ships with a trigger so it never drifts (both scaffolded in `templates/documentation-bundle.template.md`, the team picks one):
- **Git hook** (`post-commit` or `pre-push`): detects changed public surfaces/contracts/architecture in the commit, runs `/document --changed`, regenerates the affected API entries + diagrams, rebuilds `_site`, **refreshes the touched `docs-index.js` entries**, and stages the refreshed bundle.
- **CI job** (`.github/workflows/docs.yml`, on push): regenerates the bundle for the change and either commits it back or **fails the build if docs are stale** (the strict gate — the team's choice, recorded in `docs/index.md`).
The Steward decides *what changed and what must regenerate*; the hook/CI is only the trigger. **The index is inside the freshness contract**: a `docs-index.js` that disagrees with frontmatter fails the same gate as stale docs, and the gate also reports artifacts past their **`review-by`** SLA (V13) — time-based decay, not just change-based (`docs-graph.py freshness --gate fail` is the CI-able check).

## Output artifact
`docs/` (markdown bundle) + `docs/docs-index.js` (the full-sweep graph index) + `docs/index.html` (the Docs Explorer — the map) + `docs/_site/index.html` (the close-up bundle view) + the freshness hook/CI, containing: API reference (JavaDoc-style, per public member), the four diagram families as Mermaid, the architecture overview, the MoC index, and the coverage + confidence metadata. Committed alongside code.

## Definition of done (exit gate)
- [ ] Every **public/exported** member is documented or recorded as a known gap (no fabricated prose).
- [ ] All four diagram families present for the documented scope and **each node/edge verified against the code** (no aspirational diagrams).
- [ ] Every example **compiles/runs** as written (Test Architect accepts).
- [ ] All three surfaces emitted: committed markdown, the `_site` close-up view, **and** the regenerated full-sweep `docs/docs-index.js` + `docs/index.html` (the Explorer opens and every view renders).
- [ ] **Threat model + privacy review current**: both exist with `documents` links to architecture + designs, registers regenerated via `rollup`, boundary map matching the C4 Context view, missing per-design analyses recorded as gaps, resolved flags cleared.
- [ ] **Every artifact under `docs/` has valid frontmatter** (id, type, owner, review-by, summary, registry-conformant links) and a **derived** index entry that matches it; entries for deleted files removed; orphan, `review-by`-stale, and `review-suggested` nodes reported as findings (V10/V12/V13/V16); decision notes indexed (V17).
- [ ] `_meta.json` records the documented commit, coverage %, and confidence labels.
- [ ] The after-commit hook **or** CI freshness job is installed.
- [ ] Adversarial gate passed; doc-vs-code contradictions resolved or recorded; the Steward did not self-clear.

## Documentation & discoverability (last action)
For `/document` the Discoverability Mandate (V10) is the **full sweep**, already performed in Stage 5: every artifact in the repo indexed with typed links, stale entries removed, the Explorer present and loading, orphans reported. Other skills update their own entry; `/document` is where the **whole graph** gets reconciled — treat any file-without-entry, entry-without-file, or unlinked artifact discovered here as a finding to fix in this same change (V11). **Propagate impact (V16):** if this change is material (a contract, requirement, decision, public shape, or proven claim changed), traverse the **inbound** edges and push `review-suggested: { by, on, reason }` into each inbound neighbor's frontmatter (and its index entry) in this same change. **Capture session exhaust (V17):** any mid-session decision, discovered assumption, or resolved question below ADR weight becomes a linked **decision note** (`docs/notes/`, from `templates/decision-note.template.md`) before close — promote to an ADR if it later bears load. Work that is not discoverable in the Explorer is not done.

**Audit (last action).** Append an audit-log entry for this run — `python3 docs/ai-forward-pack/scripts/audit-log.py append --shortname "document-<scope>" --session "<id>" --skill document --kind skill --prompt "<the prompt, verbatim>" --summary "<what was documented>" --artifact docs/index.md` — per the Audit Mandate (`knowledge/audit-and-change-log.md`, AL5). A run that left no trace in `docs/audit/` is, like an un-indexed artifact, not done.

**Handoff:** the bundle is the input to onboarding, review, and `/define-architecture` (the layered/component diagrams become the architecture of record). Runs continuously thereafter via the freshness trigger.
