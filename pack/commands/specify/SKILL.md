# Skill: /specify

Produce a **specification** — the *what* and *why*, never the *how* — that is grounded in evidence and sharp enough to test. This is the macro-loop's first phase (BoK §IV.1; Rules of the Road Phase 4) and the unit of governance everything downstream answers to (BoK §VII.7). The output is a committed artifact, not a throwaway prompt.

**Spine:** runs the Rigor Protocol (`knowledge/rigor-protocol.md`) on the *problem*. **Authority:** the **Specification Standards** (`knowledge/specification-standards.md`, S1–S18) define the spec's **three layers** — Functional (what & why), UX (how it *works*), UI (how it *looks*) — their bottom-up dependency, and their separate owners; the **UI & Interaction Design Standard** (`ui-interaction-design.md`, U1–U20) governs the UI layer's excellence, and the **UI Archetype Grammar** (`ui-archetype-grammar.md` + `ui-archetype-catalog.md`) selects the UI's archetype as a determinism control. **Mode:** Peer Mode for authoring, Adversary Mode at the gate (`knowledge/collaborative-personas.md`).

## Grounding (first action)
Load what already exists for this problem and treat it as the **authoritative source of truth** (Rigor Protocol Stage 0; BoK §III.1): any existing spec for this area (`docs/specs/`), the domain knowledge bases (`docs/knowledge/`), and the project's governance docs. Don't re-spec something already specified without reconciling against it; if your new framing conflicts with an existing spec or a knowledge base, **surface the conflict** rather than quietly overriding it. Prefer **graph traversal** for this grounding (`knowledge-visualization.md` V15): start from this task's artifact(s) in the knowledge graph and follow the typed edges 1–2 hops (upstream `implements`/`refines`/`depends-on`, downstream `tested-by`/`documents`, and `uses-term` into the glossary), citing the traversal path; a missing edge, stale node, or orphan found here is a finding to surface. Skip this only if the user explicitly tells you not to consult prior artifacts.

## Input
A prompt, an idea, a problem statement, or a backlog item. May be one sentence. The skill's job is to expand it into a frame, narrow it with evidence, and converge on a testable spec.

## Cast
- **Peers (author together):** Orchestrator, **Product Strategist** (lead, owns the **Functional** layer), **Domain Researcher**. **When the change has a user-facing surface, add the UX Researcher / Information Architect** (owns the **UX** layer — personas, IA, user flows, wireframe-level structure). **When there is a visual UI, add UX & Accessibility** (owns the **UI** layer). Add the **Security & Identity Architect** as a peer if the problem touches identity, PII, money, or an irreversible action.
- **Adversaries (attack the spec at the gate, bottom-up):** **The Simplifier** (scope, gold-plating — soft veto), **The Test Architect** (functional criteria verifiable — hard veto on unverifiable claims), **UX Researcher / IA** (UX layer: flow integrity, IA, findability, unhappy-path coverage — **UX-specification veto**, if user-facing), **UX & Accessibility** (UI layer: state completeness, tokens, WCAG 2.2 AA — veto, if visual UI), **Security & Identity Architect** (mandatory if identity/PII/data — hard veto).

## Flow (Rigor Protocol, specialized to the problem)

**Stage 0 — Interdict the rush.** Do not propose a solution, a UI, or an architecture. If the input already contains a solution, extract the *underlying problem* it is trying to solve and spec that. (Anti-pattern: solution masquerading as requirement.)

**Stage 1 — OPEN.** Free-recall the problem space: who is this for, what are they actually trying to accomplish, what are the candidate framings of the problem, what scenarios exist (baseline / adversarial / wildcard user), what is explicitly *out*. Sketch the system the product lives in. List the unknowns (user facts, domain facts, comparables) to establish.

**Stage 2 — INTERROGATE.** Drill the frame with precise questions (Precision Questioning types): *Clarification* — what exactly is in scope; *Assumption* — what are we taking as given about the user and the value; *Evidence* — what weighs for and against this being the right problem; *Significance* — is this the problem that matters. Surface every assumption; mark each to-verify or accepted.

**Stage 3 — EVIDENCE.** Establish, don't assert (Domain Researcher leads): gather **industry comparables** (what do existing products do for this problem — named, with sources) and **user reference** (the real need, the workflow, the evidence — named). Each comparable/user claim is labeled Verified / Inferred / Flagged in the confidence ledger. Walk the **Engineering Governance checklist** (`engineering-governance.md`) and mark which SDLC lenses apply (quality attributes, threat model, privacy, accessibility, performance, release/rollback, observability) — a lens that applies but has no answer is a gap to close. **If the change has a user-facing surface (Specification Standards S1; any medium), gather the UX evidence too (UX Researcher/IA leads this part):** the **real user need / job-to-be-done** evidenced rather than assumed, the **information architecture** candidates (how things group, the navigation, the labels), and the **user flows** that will have to cover not just the happy path but the alternate/error/recovery paths. **And when there is a visual UI** (UI Standard `ui-interaction-design.md` U1–U2): the **medium(s)** and authoritative platform guidelines, the **density/expertise profile**, the **experience qualities** (defining adjectives *and their opposites*), and — for AI-facing UIs — which **HAX guidelines** apply and whether oversight/trust patterns are required (U13–U15). **Select the UI archetype (the "UX template").** If the user named a UX template/archetype, use it. **If they did not, reason over the jobs-to-be-done and the user flows (Parts A/B) and choose the best-match archetype** from `ui-archetype-catalog.md` — map the dominant JTBD to the archetype family (a guided/assessment/onboarding job → the **A-series** temporal state machines; a record-management / throughput / dashboard job → the **B-series** operational; a spatial/canvas job → **C**; a feed / conversational / streaming job → **D**; an authoring/marketing job → **E**), then pick the specific row whose exemplars and codegen posture best fit, recording **why** (the JTBD→archetype rationale). Note its canonical Archetype Signature as the starting point (facet deviations noted later per grammar G9). Gather UX/UI comparables the same way as any other (named, sourced).

**Stage 4 — DISCONFIRM (the gate, bottom-up).** Switch to Adversary Mode and review the layers **from the plane beneath upward** (S2). The Simplifier attacks scope and gold-plating; the Test Architect attacks every **functional** acceptance criterion with *"what input makes this fail? if none, it asserts nothing"*; **the UX Researcher/IA attacks the UX layer** — is the need evidenced, is the IA coherent, can the user reach the goal without guessing, do the flows cover the alternate/error/recovery paths (not just the happy one)? — holding the **UX-specification veto**; **UX & Accessibility attacks the UI layer** — complete component states, tokens not arbitrary values, WCAG 2.2 AA; Security (if convened) runs STRIDE on the new surface. A settled UI layer over an incoherent UX layer is a block (Surface-before-Structure). Resolve or record every veto. **The authors do not clear their own hard veto.** Record the gate verdict (Rules of the Road §3.2).

**Stage 5 — CONVERGE.** Write the spec to `templates/spec.template.md` as **one document with three explicit layers**, authored **bottom-up** (S1–S2):
- **Part A — Functional** (always present; Product Strategist): problem (solution-independent), personas/JTBD, core scenario, explicit non-goals, **user stories with falsifiable Gherkin acceptance criteria** (happy + error paths, measurable thresholds — Engineering Governance §1), and the **ISO 25010 NFR checklist**. For AI-integrated work, name the **LOA archetype** and tier allocation (LOA Part VI).
- **Part B — UX specification** (UX Researcher/IA; present when user-facing, else **N/A — `<reason>`**): deepened personas, **information architecture**, **user flows drawn as Mermaid flowcharts covering happy + alternate + error + recovery paths**, and the **wireframe-level structure** — plus UX acceptance criteria ("user reaches checkout in ≤ 2 steps"; "every flow has a specified recovery path"). This layer **gates Part C**.
- **Part C — UI specification** (UX & Accessibility; present when visual UI, gated behind a settled Part B, else **N/A — `<reason>`**): **records the chosen UI Archetype Signature** (`ui-archetype-grammar.md`, from the nearest `ui-archetype-catalog.md` row, with facet deviations noted — the determinism selector; **when the archetype was auto-selected because the user named no UX template, record the one-line JTBD→archetype rationale here**) and is specified to **U1–U20** — medium + platform guidelines, token references, key screens with the **complete component state set** (incl. empty/loading/error), motion + reduced-motion, real in-voice copy, **WCAG 2.2 AA** + performance budget, and for AI UIs the named **HAX** guidelines and **Shape-of-AI** patterns — each as a **falsifiable acceptance criterion** ("the empty state guides the user to first action"; "all interactive targets meet contrast AA"; "the AI shows an action plan before any irreversible step").

State the residual risk and the flagged unknowns. Traceability holds: each UX flow realizes a functional story; each UI screen realizes a UX flow; IA labels seed the glossary (S10). **Tell the user what was chosen:** when the UI archetype (UX template) was **auto-selected** — the user did not name one — your closing summary must state *which* archetype was chosen and the *JTBD-based reason*, so the choice is visible and the user can confirm or override it (it is already recorded in Part C).

## Output artifact
`docs/specs/<feature>.md` (from `templates/spec.template.md`), as **one document with three layers**: **Part A — Functional** (problem, scope, personas, core scenario, non-goals, user stories + Gherkin criteria, ISO 25010 NFRs), **Part B — UX specification** (personas, IA, user flows, wireframe structure — or N/A with reason), **Part C — UI specification** (specified to U1–U20 — or N/A with reason), plus comparables/evidence, flagged risks, and the gate record. If a layer grows large, it may be promoted to a linked companion (`docs/specs/<feature>-ux.md`) with a `documents` link back (S6). Committed alongside code.

## Definition of done (exit gate)
- [ ] **All three layers present, or each absent one marked N/A — `<reason>`** (S1); authored bottom-up, UI layer not settled before the UX layer beneath it (S2).
- [ ] Core scenario named; explicit non-goals stated.
- [ ] Every **functional** acceptance criterion is **falsifiable** and could be traced to a test (Test Architect would accept it); user stories carry Gherkin criteria covering happy + error paths; **ISO 25010 NFR checklist** walked.
- [ ] **UX layer (if user-facing):** evidenced personas, coherent IA, and **user flows covering happy + alternate + error + recovery paths drawn as flowcharts** — UX-specification veto cleared (S6–S7).
- [ ] **UI layer (if visual):** the **UI Archetype Signature** is recorded (nearest catalog row, deviations noted — `ui-archetype-grammar.md` G13) and the layer is specified to `ui-interaction-design.md` — medium, tokens, complete component states, motion, real copy, WCAG 2.2 AA, performance budget, AI-UX patterns where relevant (S8). **If no UX template was specified, the best-match archetype was auto-selected from the JTBD, its rationale recorded in Part C, and the choice surfaced in the summary** (so the user can confirm or override).
- [ ] Comparables and user claims are **sourced and labeled** (no Confident Guess).
- [ ] Governance lenses walked; applicable-but-unanswered ones flagged.
- [ ] Adversarial gate passed; vetoes resolved or recorded; authors did not self-clear.
- [ ] Confidence ledger started; residual risk stated.

## Documentation & discoverability (last action)
Per the **Knowledge Visualization & Docs Explorer Standard** (`knowledge/knowledge-visualization.md`, the Discoverability Mandate V10): after producing the spec (`docs/specs/<feature>.md`), **write the artifact's frontmatter** (the record, V2: id, title, type, status, **owner**, phase, tags, **typed links** per the relation registry, **review-by** per the type's SLA, and a real 1–3-sentence summary) and **sync the derived `docs/docs-index.js`** by running the script bundle — `python3 docs/ai-forward-pack/scripts/docs-graph.py derive` (and `flag --changed <id> --reason …` for V16 propagation) — never by generating ad-hoc scripts (V18); frontmatter wins wherever the two disagree. Ensure `docs/index.html` (the Docs Explorer) exists — instantiate it from `templates/docs-explorer.template.html` if missing — and verify the new entry has at least one typed link into the graph (an orphan is a finding). Index and diagrams land **in the same change** as the content (V11). **Propagate impact (V16):** if this change is material (a contract, requirement, decision, public shape, or proven claim changed), traverse the **inbound** edges and push `review-suggested: { by, on, reason }` into each inbound neighbor's frontmatter (and its index entry) in this same change. **Capture session exhaust (V17):** any mid-session decision, discovered assumption, or resolved question below ADR weight becomes a linked **decision note** (`docs/notes/`, from `templates/decision-note.template.md`) before close — promote to an ADR if it later bears load. Work that is not discoverable in the Explorer is not done.

**Audit (last action).** Append an audit-log entry for this run — `python3 docs/ai-forward-pack/scripts/audit-log.py append --shortname "specify-<subject>" --session "<id>" --skill specify --kind skill --prompt "<the prompt, verbatim>" --summary "<what it produced>" --artifact docs/specs/<feature>.md` — per the Audit Mandate (`knowledge/audit-and-change-log.md`, AL5); a run that left no trace in `docs/audit/` is, like an un-indexed artifact, not done. When the spec settles a load-bearing decision, also add a change-log entry (`audit-log.py change`, CL1).

**Handoff:** → `/define-architecture` (for new/load-bearing systems) or → `/design` (for a feature within an existing architecture).
